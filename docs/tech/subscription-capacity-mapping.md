# 订阅容量反推与账号-订阅映射（1 号 ↔ 1 group ↔ N 用户/key）

status: design / 草案 v2
scope: backend (account scheduling, billing, subscription, rate limit) + frontend (subscription view)

> v2 变更：放弃 v1 的「多账号池化 + ΣC 池顶」模型。改为 **1 上游号 ↔ 1 订阅 group**，
> 一个号被多个用户订阅、在 key 间均分；**每个 api_key 钉死单号**（请求永不跨号），
> 因此每个 key 永远只面对一个号的一套 5h/7d 窗口——**重置天然对齐上游**。
> 展示用方向 2：每个用户只看到自己那一份，不感知共享人数与号的真实经济。

## 1. 背景与目标

网关上游是 Anthropic 订阅号（Pro / Max 5x/10x/20x），每个号有「按周 + 按 5h + 按用量占比」的额度，且各窗口有上游自己的重置时刻。目标：

1. 不靠人工配置，动态反推每个号的真实 5h/7d 容量并自动定档。
2. 把「一个号」做成「一个订阅产品」，多个用户订阅、在各自 key 间均分该号容量。
3. 订阅的限额与重置 **忠实透传绑定号的真实 5h/7d 窗口**（不造独立窗口）。
4. 每个用户的视图是私有的：只见自己那份用量与份额，不知道号分给了几个人。

## 2. 核心模型

```
1 个上游号 (009, Pro)
  └─ 1 个订阅 group   (1:1 绑定，group 只绑这一个号)
       ├─ 用户 A 订阅 ── key a1, a2 ...
       ├─ 用户 B 订阅 ── key b1 ...
       └─ 用户 C 订阅 ── key c1 ...
     号容量按分配策略切给订阅（当前=均分硬顶，份额=容量/N，不可借用）
```

两条铁律：

1. **1 订阅 group : 1 号**（group 只绑一个号）。
2. **1 api_key : 1 号**（group 只有一个号 → 组内任何 key 的请求必打这个号，永不跨号）。

由铁律 2 推出的关键性质：**每个 key 只面对一个号的一套 5h/7d 窗口与重置**，所以
「重置对齐」天然成立——不会出现 v1 多账号池里「各号窗口各自重置、对不齐」的问题。
共享只发生在「一个号 vs 组内多个 key」这一层（均分），key 从不跨号。

扩容方式：**加号 = 加订阅 group**（每个 group 绑一个新号），不把多个号并进一个池。

## 3. 容量反推（号的真实窗口额度从哪来）

上游每个响应头携带占比信号，网关已被动采样：

- `anthropic-ratelimit-unified-7d-utilization`（7d 占比 0~1）
- `anthropic-ratelimit-unified-5h-utilization`（5h 占比 0~1）

反推（分母用标准成本 total_cost，对齐上游 API 等价口径；阶段一已实现并部署）：

```
号窗口容量 = 窗口内标准成本 / 窗口 utilization     （utilization < 门槛则跳过，无信号不写）
EWMA 平滑 + log 空间就近吸档 {pro/max5x/max10x/max20x}
```

prod 实测（009，oauth=Pro，user 已确认档位）：7d $21.03/0.11 → **$191/周**；5h $4.51/0.18 → **$25/5h**。

写回 `account.extra`：`inferred_capacity_7d` / `inferred_capacity_5h` / `inferred_tier` /
`inferred_tier_confidence` / `inferred_capacity_updated_at`。`inferred_` 前缀已登记
scheduler-neutral，写它不触发调度（阶段一只读观测的物理保证）。

## 4. 窗口透传 + 重置对齐（v2 核心）

订阅的限额与重置 **= 绑定号的真实 5h/7d 窗口**，不再用网关独立的 weekly 窗口：

| 维度 | 来源 | 现状 |
|---|---|---|
| 5h 限额 | 号反推 `inferred_capacity_5h`（×安全系数） | 反推已有 |
| 5h 重置 | 号 `session_window_end`（已持久化） | **已有** |
| 7d 限额 | 号反推 `inferred_capacity_7d`（×安全系数） | 反推已有 |
| 7d 重置 | 上游 `anthropic-ratelimit-unified-7d-reset` 头 | **需补持久化**（当前只在 429 时读，未存） |

数据缺口只有一处：**7d reset 需在被动采样时一并存到号**（和 util 一起，小改动）。5h reset
已有（`session_window_end`）。

安全系数（如 ×0.85）留给反推误差 + 突发毛刺，避免把号打到上游真 429。

## 5. 分配策略（一个号在多订阅间分配）

号容量按**可插拔的分配策略**切给订阅，5h 与 7d 各一套。**当前唯一策略 = 均分**：

```
份额 = 号窗口容量 / N          每个订阅一份等额硬顶
不可借用：用满自己份额即止，不蹭别人的闲置额度
撞自己份额上限 → 429（沿用 ErrAPIKeyRateLimit5h/7dExceeded），客户端重试/等重置
```

**硬顶、不回收**（明确否决 max-min 的「借闲置」——它会让份额视图出现「已用>份额」，
也让单用户可用量受不可见的他人影响，违背方向 2 私有视图）。

**策略是抽象、可换**：均分只是第一个实现。未来可加 加权（按付费档）/ 优先级 / 阶梯 等，
走同一接口插拔；现在不做，只留扩展位，别把均分逻辑写死进准入主干。

复用现有 `api_key.usage_5h/usage_7d` 追踪 + `billing_cache_service.evaluateRateLimits`
注入点（阶段一已确认）。N 的口径（动态按活跃订阅数重算 vs 固定售卖槽位）见 §9。

## 6. 展示（方向 2：私有视图）

每个用户的订阅页 **只见自己那一份**：

**显示**：我的本窗口用量、我的份额（硬顶）、重置时间（= 号真实重置，对用户呈现为"我的重置"）。
**隐藏**：号的总容量、共享人数 N、"这是共享号"这一事实、号的真实经济数字（如 $191）。

效果：用户感觉像独享一份订阅，不感知背后是和别人共享一个号。份额硬顶、不可借用，
所以"已用 / 份额"始终是干净的进度条，不会出现已用超过份额。

**铁律：所有用户可见字段都是产品文案。** 订阅卡片的 group `name` / `description`、套餐名、
单位、提示语全部直接渲染给用户——禁内部黑话（如「容量反推」「订阅型 group」「绑 009」
「pilot」），禁暴露号经济数字。配置这些字段时按"给用户看"写，不是内部备注。见
[[feedback_no_tech_language_in_user_ui]]。

（份额硬顶、不可借用——这条已定，不再是开放点。）

## 7. 现有地基 + 扩展点

| 能力 | 现状 | v2 要做 |
|---|---|---|
| utilization 采样 + 反推/定档 | 阶段一已上线（写 inferred_*） | 复用 |
| 5h 窗口/重置（session_window_end） | 已持久化 | 复用 |
| 7d 重置 | 仅 429 时读 | **补**：被动采样时持久化 7d reset |
| 订阅层 UserSubscription | 有 daily/weekly/monthly 窗口 + 网关自定重置 | **改**：加 5h 窗口；限额/重置改由绑定号驱动，弃网关独立 weekly |
| group 限额强制（ValidateAndCheckLimits） | 用 group.weekly_limit_usd + 网关窗口 | **改**：限额=号反推容量，窗口=号真实窗口 |
| per-key 限额（evaluateRateLimits） | 静态 rate_limit | **改**：分配策略（当前=均分硬顶，份额=号容量/N，不可借用），策略可插拔 |
| group↔account 多对多 | 支持 | **约束**：订阅 group 只绑 1 个号 |
| 用户侧 /subscriptions 页 | pilot 已恢复 | **改**：方向 2 私有视图（5h+7d 两窗口，隐藏共享，产品文案） |
| fallback 字段 | 非配额触发（客户端降级 / 无效请求改道） | 不复用为配额溢出 |

**v1 删除项**：多账号池化、ΣC 池顶、跨号聚类自标定的池容量、安全系数作用于"池"——
这些在 1:1 模型下不存在（号即订阅，无池）。聚类自标定仍可用于"无锚点时推断号档位"，
但不再用于池容量。

## 8. 实施阶段

1. **阶段一（已上线）**：反推 + 定档，写 inferred_*，零 enforcement 影响。
2. **阶段二（窗口透传）**：补 7d reset 持久化；订阅限额/窗口/重置改由绑定号驱动（5h+7d）；
   /subscriptions 方向 2 私有视图（显示两窗口、隐藏共享、产品文案）。先单 group（009）灰度。
3. **阶段三（分配准入）**：evaluateRateLimits 接分配策略接口，落地均分硬顶（份额=号容量/N，
   不可借用），撞份额 429。策略做成可插拔（未来加权/优先级走同一接口）。多用户共享同一号实测。
4. **阶段四**：observe 稳定后推广；加号即加 group 扩容。

## 9. 开放点 / 风险

- **N 的口径**：均分分母按「动态活跃订阅数重算」还是「固定售卖槽位」？前者利用率高但用户份额
  会随他人加入而缩水（方向 2 下用户莫名变少）；后者份额稳定但需管理槽位。实现前定。
- **安全系数取值**：5h/7d 各留多少 headroom 给反推误差 + 突发。
- **冷号**：无流量则容量/档位 unknown，订阅无法定额——需保守默认或主动小流量探测。
- **多用户同号互相影响**：硬顶下各订阅份额独立、互不挤占（不借用），但号总容量被瓜分，
  份额 = 容量/N 本身就比独享小；这是共享换供给的固有代价。
- **号被上游降级/封**：绑定号出问题会直接影响其上所有订阅用户；1:1 无池兜底（这是 1:1
  换"对齐+诚实"的代价，与池化的"多号容灾"相反）。
- **号被上游降级/封**：绑定号出问题会直接影响其上所有订阅用户；1:1 无池兜底（这是 1:1
  换"对齐+诚实"的代价，与池化的"多号容灾"相反）。
